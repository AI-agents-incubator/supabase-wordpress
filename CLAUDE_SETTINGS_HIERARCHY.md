# Claude Code — Иерархия настроек (settings hierarchy)

## Принцип

Каждый следующий уровень **полностью перекрывает** предыдущий.
Нижний уровень всегда побеждает.

---

## Уровни (от низшего к высшему приоритету)

```
1. ~/.claude/settings.json           ← глобальный базовый
2. ~/.claude/settings.local.json     ← глобальный локальный
3. .claude/settings.json             ← проектный (в git, для команды)
4. .claude/settings.local.json       ← проектный локальный (не в git, только для тебя)
```

Уровень 4 перекрывает уровень 3, который перекрывает уровень 2, который перекрывает уровень 1.

---

## Что происходит при отсутствии файлов

| Ситуация | Результат |
|----------|-----------|
| Нет `.claude/` в проекте вообще | Работают только глобальные настройки (`~/.claude/`) |
| Есть `.claude/settings.json`, нет `.local.json` | Проектный файл перекрывает глобальный |
| Есть оба проектных файла | `settings.local.json` перекрывает всё |

---

## Практическое применение

**Глобальный (`~/.claude/settings.json`):**
Базовые разрешения для всех проектов. Здесь имеет смысл давать широкую автономию (`Bash(*)`), чтобы не прерываться на рутинные операции в любом проекте.

**Проектный (`[project]/.claude/settings.json`):**
Специфика проекта, которая нужна всей команде. Коммитится в git. Например: разрешённые WebFetch-домены, специфические Skills.

**Проектный локальный (`[project]/.claude/settings.local.json`):**
Твои личные настройки для этого проекта. **Не коммитится в git.** Используется для:
- Доступа к продакшн-серверу с подтверждением (`ask`)
- Полной автономии в экспериментальных проектах (`bypassPermissions: true`)
- Любых настроек, которые не должны попасть в репозиторий

---

## bypassPermissions

```json
{
  "permissions": {
    "bypassPermissions": true
  }
}
```

Отключает **всю** систему разрешений: allow, ask, deny — всё игнорируется.
Claude выполняет любые операции без единого вопроса.

**Работает только в `settings.local.json`** — намеренное ограничение безопасности.
Причина: если бы это работало в `settings.json` (который в git), любой репозиторий мог бы захватить полный контроль над машиной.

## Иерархия CLAUDE.md

Работает аналогично — каждый следующий уровень добавляет или уточняет контекст:

```
~/.claude/CLAUDE.md                      ← глобальный (для всех проектов)
[project]/CLAUDE.md                      ← проектный корневой
[project]/src/CLAUDE.md                  ← для конкретной папки
[project]/src/components/CLAUDE.md       ← ещё глубже
```

Все найденные файлы **объединяются** (не перекрывают друг друга полностью) — Claude видит все уровни одновременно. Более глубокий файл имеет больший вес для кода в своей директории.

Полезно когда у разных модулей своя специфика: `src/api/` — REST-контракты, `src/db/` — правила миграций.

---

## settings.json vs CLAUDE.md — разные уровни контроля

Это **не одно и то же**, несмотря на похожую иерархию:

| | `settings.json` | `CLAUDE.md` |
|---|---|---|
| Где обрабатывается | Приложение (Node.js runtime) | Модель (системный промпт) |
| Часть промпта? | **Нет** | **Да** |
| Тип ограничения | Жёсткий (код) | Мягкий (инструкция) |
| Модель может обойти? | Нет | Теоретически да |

`settings.json` — **охранник на входе**: решает до того, как модель вообще что-то видит.
`CLAUDE.md` — **инструкция внутри**: модель читает и следует ей, пока охранник не запретил.

**Как разрешаются противоречия:**

| Случай | Результат |
|--------|-----------|
| CLAUDE.md говорит "спрашивай перед push", settings.json разрешает `Bash(git push*)` | settings.json побеждает — приложение не спросит |
| CLAUDE.md говорит "не удаляй файлы", settings.json разрешает `Bash(rm*)` | CLAUDE.md побеждает — модель откажется вызвать инструмент |
| settings.json запрещает через `deny` | settings.json всегда — модель не узнает, что инструмент существует |

**Вывод:** `deny` в settings.json — абсолютный запрет. `allow` — разрешение, но модель всё равно может отказаться на основе CLAUDE.md. Противоречия управляются архитектурой, не моделью.

## Механика prompt injection и реальные риски

Модель **никогда не видит** settings.json. Она получает только промпт и возвращает намерение (structured tool call). Runtime перехватывает это намерение и сверяет с settings.json.

```
Промпт → Модель → tool call (намерение) → [Runtime проверяет settings.json] → выполнить / отказать
```

Поэтому:

```
deny Bash(rm*)   +  любой промпт "удали всё"  →  не выполнится никогда
allow Bash(*)    +  вредоносный промпт         →  выполнится, если модель согласилась
```

**Prompt injection** (вредоносный текст в данных, который пытается изменить поведение модели) не может обойти `deny`. Но при широком `allow` или `bypassPermissions` — охранник пропустит всё, что модель решила сделать. Ответственность переносится на качество инструкций в CLAUDE.md и здравый смысл модели.

## Философия автономии: свобода внутри границ

Парное программирование с постоянными подтверждениями неэффективно — оно прерывает поток и требует, чтобы человек занимался только программированием. Агент достаточно мощный для автономного выполнения задач высокого уровня.

Но автономия имеет реальные риски не столько от внешних угроз, сколько от самой модели:
- сломать код, который сама же написала
- несогласованно изменить продакшн
- натворить бед в файловой системе

**Рабочий паттерн — sandbox по проекту:**

Дать модели полную свободу внутри папки проекта, жёстко ограничив выход за периметр. Модель делает всё что угодно с файлами проекта — но продакшн-доступ требует явного подтверждения.

| Тип проекта | Рекомендуемая настройка |
|-------------|------------------------|
| Эксперимент / R&D | `bypassPermissions: true` в `settings.local.json` |
| Рабочий проект без продакшна | `Bash(*)` в глобальном, ничего дополнительного |
| Проект с продакшн-деплоем | `Bash(*)` глобально + `ask` для SSH/DB в `settings.local.json` |
| Критичный продакшн | `deny` для деструктивных операций в `settings.json` проекта |

**Ключевой принцип:** `deny` — единственная настоящая защита. Всё остальное — договорённость с моделью. Чем важнее защищаемый ресурс, тем больше смысла использовать `deny`, а не полагаться на CLAUDE.md.
