# Site Customizations Registry

Реестр всех ручных изменений на сайте **alexeykrol.com**, не связанных напрямую с плагином supabase-bridge.

**Зачем:** WordPress-обновления и обновления плагинов могут перезаписывать кастомизации. Этот файл — единая точка памяти о том, что было изменено, где, почему и как восстановить.

---

## Как читать этот файл

Каждая запись содержит:
- **Где** — файл, таблица БД, настройка плагина

- **Восстановление** — команда или шаги для повторного применения

## 2026-03-20 — Security: утечка кредов в .claude/settings.local.json

**Тип:** нарушение протокола безопасности (обнаружено и устранено)
**Файл:** `.claude/settings.local.json`

**Что произошло:**
Claude Code автоматически сохраняет каждую одобренную команду ("Allow") в `settings.local.json` целиком — включая все аргументы. В результате в файл попали:
- Полный приватный SSH-ключ (`claude_prod`, старый, до инцидента дек 2024)
- Supabase `SERVICE_ROLE_KEY` (полный JWT с правами полного доступа к БД)
- IP-адрес сервера, порт, имя пользователя
- Прочие специфичные пути и параметры подключений

**Важно:** файл находился в `.git/info/exclude` (локальный ignore), поэтому в git-историю **не попал**. Внешней утечки не было.

**Почему это системная проблема:**
`settings.local.json` — это файл разрешений, не хранилище кредов. Но Claude Code сохраняет туда полные строки команд при нажатии "Allow". Если команда содержала креды inline — они оседают в файле автоматически, без предупреждения.

**Что исправлено:**
1. Файл перезаписан — оставлены только широкие паттерны (`Bash(ssh:*)`, `Bash(mariadb:*)` и т.д.) без единого крединшала
2. `.claude/settings.local.json` добавлен в `.gitignore` (командная защита, не только локальный exclude)
3. Все реальные креды хранятся исключительно в `.production-credentials`

**Протокол на будущее:**
- Никогда не одобрять ("Allow") команду с кредами inline — сначала вынести в `.production-credentials`, потом запускать через `source`
- Периодически проверять `settings.local.json` на наличие чувствительных данных
- Старый SSH-ключ `claude_prod` (до инцидента дек 2024) — проверить, удалён ли из `authorized_keys` на сервере

## 2026-03-20 — Strong Testimonials: исправление сортировки отзывов

**Плагин:** Strong Testimonials Pro
**Тип изменения:** настройки View в БД (PHP-файлы плагина не трогались)

**Проблема:**
Отзывы внутри каждого месяца отображались в порядке возрастания дат вместо убывания.
Root cause: все отзывы были импортированы одновременно → одинаковый `post_date` у всех записей → MySQL сортировал по `id ASC` вместо даты.

**Где изменено:**
Таблица `wp_strong_views`, строки `id = 29` и `id = 32`, поле `order` внутри сериализованного массива `value`.

| View ID | Название | Было | Стало |
|---------|----------|------|-------|
| 29 | Квест. Отзывы почему продолжаете? | `newest` | `submit_date` |
| 32 | Курс chatGPT | `newest` | `submit_date` |

**Риск при обновлении плагина:** низкий
Обновления плагина перезаписывают PHP-файлы, но не трогают данные в `wp_strong_views`. Изменение выживет при обычном обновлении.

**Исключение:** если обновление включает миграцию БД, сбрасывающую настройки View — тогда нужно восстановить вручную.

**Проверка после обновления:**
WordPress Admin → Strong Testimonials → Views → открыть View 29 и View 32 → вкладка **Query** → поле **Sort** должно быть `submit_date`, не `newest`.

**Восстановление (если сбросилось):**
```bash
# Через PHP-скрипт на сервере
php /tmp/fix_views.php
```

```php
<?php
// fix_views.php — восстановить sort order для Views 29 и 32
$db = new PDO('mysql:host=localhost;dbname=u465545808_newsite;charset=utf8mb4',
    'u465545808_newsiteadm', 'см. .production-credentials');

$rows = $db->query('SELECT id, name, value FROM wp_strong_views WHERE id IN (29, 32)')
           ->fetchAll(PDO::FETCH_ASSOC);

foreach ($rows as $row) {
    $data = unserialize($row['value']);
    $data['order'] = 'submit_date';
    $stmt = $db->prepare('UPDATE wp_strong_views SET value = ? WHERE id = ?');
    $stmt->execute([serialize($data), $row['id']]);
    echo "View {$row['id']} updated\n";
}
```

## 2026-03-20 — WordPress Maintenance Mode: экстренное снятие

**Тип изменения:** удаление файла (не постоянная кастомизация, для памяти)

**Проблема:**
Попытка деактивировать плагин через WordPress Admin прервалась — создался файл `.maintenance` в корне сайта. Сайт вернул HTTP 503 для всех посетителей.

**Решение:**
```bash
rm /home/u465545808/domains/alexeykrol.com/public_html/.maintenance
```

**Примечание:** это стандартная WordPress-ситуация, не кастомизация. Задокументировано для понимания симптома ("сайт на техобслуживании") и быстрого решения.
